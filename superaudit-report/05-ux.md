# 6. UX funcional

> Auditor: ux-auditor | Fecha: 2026-04-23

## Resumen ejecutivo

Brew-TUI v0.2.0 presenta una arquitectura de flujos coherente y bien estructurada, con patrones consistentes de loading/error, confirmaciones en acciones destructivas masivas, y navegacion global predecible mediante atajos de teclado. Sin embargo, persisten cuatro hallazgos no conformes: la correccion de "Esc dismisses ProgressLog" introducida en InstalledView no se aplico a OutdatedView ni a SearchView, dejando al usuario atrapado en la pantalla de progreso completado; el boton "Open Brew-TUI" de BrewBar sigue hardcodeando Terminal.app contrariamente a lo declarado como correccion en v0.2.0; y BrewBar no solicita confirmacion en la accion individual de upgrade por paquete. En BrewBar la correccion de licencia expirada funciona correctamente (degraded mode, no termina). La calidad UX general es alta para una herramienta TUI/menubar, con copy claro, feedback de progreso en tiempo real, y cobertura casi total de estados de error.

## 6.1 Flujos criticos

### Checklist

* [ ] Onboarding — **Baja**: No existe flujo de onboarding. El usuario inicia directamente en el Dashboard sin ningun tutorial, contexto de primeros pasos ni explicacion de los atajos de teclado. Aceptable para una herramienta tecnica dirigida a desarrolladores, pero genera friccion en primeros usos.
* [x] Login / Autenticacion — No aplica para TUI. La activacion Pro es via CLI (`brew-tui activate <key>`) y no es un flujo interactivo dentro de la TUI en si.
* [x] Logout / Deactivate — Conforme. La vista Account ofrece `d: deactivate` con `ConfirmDialog` previo.
* [ ] Registro / Signup — **Baja**: No existe flujo de registro en la TUI. El usuario debe adquirir licencia externamente (polar.sh) y activarla via CLI. El `UpgradePrompt` muestra la URL y el comando, pero no hay flujo guiado.
* [ ] Recuperacion de contrasena — **No aplica**: Modelo de licencia por clave, no por usuario/contrasena.
* [x] Primer valor util del producto — Conforme. El Dashboard carga automaticamente al inicio con estadisticas de formulae, casks, outdated y servicios. El valor central es inmediatamente visible.
* [x] Crear elemento (perfil) — Conforme. ProfilesView ofrece flujo `n` -> nombre -> descripcion con TextInput y navegacion entre pasos.
* [x] Editar elemento — Conforme. ProfilesView ofrece edicion de nombre y descripcion de perfil existente mediante `e` desde el panel de detalle.
* [x] Borrar elemento — Conforme. InstalledView (`u`), ProfilesView (`d`) y SmartCleanupView (`c`) confirman antes de borrar mediante `ConfirmDialog`.
* [x] Confirmacion de acciones criticas — Conforme en general: upgrade individual/masivo (OutdatedView), uninstall (InstalledView), stop/restart servicios (ServicesView), cleanup (SmartCleanupView), import de perfil (instalacion masiva), deactivate licencia. La unica excepcion es `start` de servicio (ejecuta sin confirmacion, evaluado en 6.2).
* [ ] Esc dismisses ProgressLog (post-operacion) — **Media**: El patron fue corregido en InstalledView pero no en OutdatedView ni SearchView. Tras completar un upgrade o una instalacion en esas vistas, el usuario no puede volver a la lista de resultados sin salir completamente de la vista.
* [x] Sincronizacion / refresh manual — Conforme. OutdatedView (`r`), DoctorView (`r`), ServicesView (`r`), SecurityAuditView (`r`), SmartCleanupView (`r`), y el boton de refresh en BrewBar popover.

## 6.2 Criterios de UX por flujo

### Flujo: Dashboard (Vista general)

* **Nombre:** Dashboard
* **Objetivo del usuario:** Obtener una vista rapida del estado del sistema Homebrew: cuantos paquetes hay, cuantos estan desactualizados, estado de servicios.
* **Inicio del flujo:** Arranque de la aplicacion (`currentView: 'dashboard'` por defecto).
* **Final esperado:** El usuario ve StatCards con formulae, casks, outdated, services, y la informacion del sistema (version Homebrew, prefix, fecha de actualizacion).
* **Pasos:**
  1. La TUI arranca y `DashboardView` se monta, disparando `fetchAll()`.
  2. Se muestra `<Loading message="Fetching Homebrew data...">` mientras `loading.installed` es `true`.
  3. Al completar la carga, se renderizan StatCards + SystemInfo + lista de outdated (hasta 10) + servicios con error.
  4. El usuario puede navegar a otras vistas via `1-0`, `Tab`, o Esc.
* **Fricciones detectadas:**
  - Dashboard no tiene atajos de accion propios (es solo lectura). Falta un hint visible de que los StatCards son navegables (pulsar `4` para ir a Outdated cuando hay paquetes desactualizados).
  - Si `errors.installed` tiene valor, se muestra `ErrorMessage` pero sin opcion de retry desde la propia vista (el usuario debe saber que `r` no esta disponible aqui).
* **Puntos de abandono:** Ninguno significativo.
* **Mensajes ambiguos:** "Fetching Homebrew data..." es claro. Los StatCards usan labels directas.
* **Feedback del sistema suficiente:** Si
* **Tiempo a valor aceptable:** Si — el primer render util ocurre en cuanto termina `fetchAll()`, tipicamente 1-3 segundos.
* **Recuperacion ante error:** Parcial — se muestra el error pero no hay accion de retry en la vista.
* **Consistencia con el resto del producto:** Si

### Flujo: Installed (Paquetes instalados)

* **Nombre:** Installed — Gestion de paquetes instalados
* **Objetivo del usuario:** Ver todos los paquetes instalados (formulae/casks), filtrar, y desinstalar.
* **Inicio del flujo:** Pulsar `2` o navegar via Tab a "Installed".
* **Final esperado:** El usuario localiza el paquete deseado, confirma la desinstalacion y ve el resultado.
* **Pasos:**
  1. `fetchInstalled()` se ejecuta al montar la vista. Se muestra spinner.
  2. Se renderizan las dos pestanas (Formulae/Casks) con cursor en posicion 0.
  3. El usuario navega con `j/k` o flechas. Puede cambiar de pestana con `f`.
  4. Pulsar `/` activa el modo busqueda (SearchInput activo, modal abierto para suprimir teclas globales).
  5. Pulsar `Enter` en un item navega a `PackageInfoView`.
  6. Pulsar `u` en un item muestra `ConfirmDialog("Uninstall X?")`.
  7. Confirmar dispara `stream.run(['uninstall', name])` y muestra `ProgressLog`.
  8. Al terminar el stream, aparece borde verde/rojo con resultado. `esc` limpia el log y refresca la lista.
* **Fricciones detectadas:**
  - El estado del filtro (`filter`, `cursor`, `tab`) se pierde al navegar a otra vista y volver. El usuario debe re-filtrar.
  - No hay indicador de scroll "virtualizado" tipo porcentaje; solo los contadores de "N more above/below".
* **Puntos de abandono:** Ninguno critico detectado.
* **Mensajes ambiguos:** Ninguno relevante. Los badges (outdated, pinned, keg-only, dep) son claros.
* **Feedback del sistema suficiente:** Si
* **Tiempo a valor aceptable:** Si
* **Recuperacion ante error:** Si — `ErrorMessage` en estado de error de carga; resultado de stream con icono de error y `esc:back` para reintentar.
* **Consistencia con el resto del producto:** Si

### Flujo: Search (Busqueda e instalacion)

* **Nombre:** Search — Buscar paquetes en Homebrew e instalar
* **Objetivo del usuario:** Buscar un paquete por nombre, ver resultados de formulae y casks, e instalar el deseado.
* **Inicio del flujo:** Pulsar `3` o navegar a "Search".
* **Final esperado:** El paquete se instala y la lista de instalados se actualiza.
* **Pasos:**
  1. El usuario escribe en el `TextInput` (foco inmediato al montar la vista).
  2. Pulsar `Enter` dispara `doSearch(query)` si `query.length >= 2`.
  3. Se muestra `<Loading message="Searching...">` mientras la peticion corre.
  4. Se renderizan los resultados divididos en "Formulae" y "Casks" (max 20 cada uno).
  5. El usuario navega con `j/k`. `Enter` navega a `PackageInfoView`. `i` muestra `ConfirmDialog`.
  6. Confirmar dispara `stream.run(['install', name])` y reemplaza la vista con `ProgressLog`.
  7. **PROBLEMA**: Al completar la instalacion, el hint dice `esc:back` pero `key.escape` no esta manejado en el bloque de stream-finalizado dentro del `useInput`. El Esc global llama a `goBack()` en lugar de limpiar el stream y volver a los resultados.
  8. El `useEffect` en lineas 54-58 refresca `fetchInstalled()` una vez que el stream termina sin error.
* **Fricciones detectadas:**
  - Query de 1 caracter pulsar Enter no da feedback: la funcion retorna silenciosamente.
  - Tras la instalacion, no hay forma de volver a los resultados de busqueda previos sin re-buscar.
  - El campo de texto es `TextInput` de `@inkjs/ui` (uncontrolled), lo que impide pre-rellenar con la query anterior al volver a la vista.
* **Puntos de abandono:**
  - Usuario con query de 1 caracter que no entiende por que no pasa nada.
  - Usuario que tras instalar no puede volver a los resultados.
* **Mensajes ambiguos:** El hint `"type to search"` en el footer es correcto pero no indica el minimo de 2 caracteres.
* **Feedback del sistema suficiente:** Parcial — falta feedback para query demasiado corta.
* **Tiempo a valor aceptable:** Si
* **Recuperacion ante error:** Si — `searchError` se muestra en rojo si la busqueda falla.
* **Consistencia con el resto del producto:** Parcial — la gestion del ProgressLog completado es inconsistente con InstalledView.

#### Checklist

* [x] El usuario entiende que hacer
* [x] El sistema comunica progreso
* [x] Hay confirmacion al completar
* [ ] Hay recuperacion al fallar — tras install completado, Esc no limpia el stream; navega fuera
* [ ] No hay pasos redundantes

### Flujo: Outdated (Actualizaciones disponibles)

* **Nombre:** Outdated — Ver y aplicar actualizaciones
* **Objetivo del usuario:** Ver que paquetes tienen actualizacion disponible y actualizarlos individualmente o todos a la vez.
* **Inicio del flujo:** Pulsar `4` o navegar a "Outdated".
* **Final esperado:** Los paquetes seleccionados se actualizan y la lista refleja el estado nuevo.
* **Pasos:**
  1. `fetchOutdated()` al montar. Spinner durante carga.
  2. Lista con nombre, version actual -> version nueva (`VersionArrow`), badge `[pinned]` si corresponde.
  3. `Enter` en un item: `ConfirmDialog("Upgrade X?")`. Confirmar: `stream.run(['upgrade', name])`.
  4. `A` (mayuscula): `ConfirmDialog("Upgrade all N packages?")`. Confirmar: `stream.run(['upgrade'])`.
  5. `p`: toggle pin/unpin del paquete bajo cursor.
  6. `r`: refresh manual de la lista.
  7. Stream en curso: `ProgressLog` visible, `esc:cancel` disponible.
  8. **PROBLEMA CRITICO**: Stream completado: `ProgressLog` permanece visible permanentemente porque el guard de render `if (stream.isRunning || stream.lines.length > 0)` (linea 69 de `outdated.tsx`) nunca se libera — no existe llamada a `stream.clear()` en ningun path post-completado. El `useInput` carece de la rama `if (!stream.isRunning && stream.lines.length > 0)` que detecta el estado finalizado. En consecuencia: (a) `Esc` ejecuta `goBack()` global en lugar de limpiar el log — el usuario sale de la vista sin querer; (b) `r` llama a `fetchOutdated()` actualizando los datos del store, pero como `stream.lines.length > 0` persiste, la vista NUNCA sale del bloque `ProgressLog` para mostrar la lista actualizada — `r` es efectivamente no funcional en este estado. El unico escape es salir de la vista (Esc) y volver a entrar.
* **Fricciones detectadas:**
  - Tras upgrade completado, el usuario esta atrapado en el ProgressLog sin ruta de salida funcional hacia la lista de outdated. Tanto `r` como `Esc` tienen comportamientos contrarios a lo que los hints visuales prometen.
  - El hint inline `r:refresh esc:back` es activamente enganoso: `r` no muestra la lista actualizada (el render sigue bloqueado) y `esc:back` no vuelve a la lista sino que navega a la vista anterior via `goBack()`.
* **Puntos de abandono:** Usuario que pulsa Esc esperando volver a la lista y en cambio sale de la vista completamente.
* **Mensajes ambiguos:** El hint inline `r:refresh esc:back` (lineas 88-89) es ambiguo y enganoso: `esc:back` implica "volver a la lista", pero en realidad "vuelve a la vista anterior" via `goBack()`.
* **Feedback del sistema suficiente:** Si
* **Tiempo a valor aceptable:** Si
* **Recuperacion ante error:** Si — borde rojo con mensaje de error en stream fallido.
* **Consistencia con el resto del producto:** No — InstalledView tiene el patron correcto; OutdatedView no.

#### Checklist

* [x] El sistema comunica progreso
* [x] Hay confirmacion al completar

### Flujo: Package Info (Detalle de paquete)

* **Nombre:** Package Info — Ver detalle e instalar/desinstalar/actualizar un paquete
* **Objetivo del usuario:** Ver toda la informacion de un paquete y operar sobre el (instalar, desinstalar, actualizar).
* **Inicio del flujo:** Pulsar `Enter` sobre un paquete en InstalledView o SearchView; navega a `package-info`.
* **Final esperado:** El usuario completa la operacion deseada o vuelve a la lista.
* **Pasos:**
  1. `api.getFormulaInfo(packageName)` al montar. Spinner con `loading_package`.
  2. Se muestra nombre, version, badges (installed/outdated/pinned/keg-only/deprecated), descripcion, detalles (homepage, license, tap, versiones), dependencias, caveats.
  3. Segun estado: `i` instalar (si no instalado), `u` desinstalar (si instalado), `U` actualizar (si outdated).
  4. Cada accion muestra `ConfirmDialog` previo.
  5. Stream en curso: `ProgressLog` con `esc:cancel`.
  6. Stream completado: resultado con icono + `esc:back`. `esc` llama a `goBack()` — correcto en este contexto.
  7. Post-stream: refresca `getFormulaInfo()` para reflejar nuevo estado instalado.
* **Fricciones detectadas:**
  - La vista no tiene atajo de numero de tecla (no esta en `VIEW_KEYS`). Solo accesible via `Enter` desde otras vistas.
  - Si se navega a `package-info` desde Search y el paquete es un cask, `getFormulaInfo()` puede fallar (la API es para formulae). No existe un path de `getCaskInfo()` equivalente.
* **Puntos de abandono:** Ninguno significativo.
* **Mensajes ambiguos:** `pkgInfo_noPackage` ("No package selected. Go to Installed and press Enter...") es claro.
* **Feedback del sistema suficiente:** Si
* **Tiempo a valor aceptable:** Si
* **Recuperacion ante error:** Si
* **Consistencia con el resto del producto:** Si

### Flujo: Services (Gestion de servicios Homebrew)

* **Nombre:** Services — Ver y controlar servicios
* **Objetivo del usuario:** Ver el estado de los servicios gestionados por Homebrew y iniciar, detener o reiniciar alguno.
* **Inicio del flujo:** Pulsar `5` o navegar a "Services".
* **Final esperado:** El servicio cambia de estado y la lista se actualiza.
* **Pasos:**
  1. `fetchServices()` al montar. Spinner.
  2. Tabla con nombre, status badge (started/stopped/error/none), usuario, exit code si aplica.
  3. `s`: inicia el servicio bajo cursor directamente (sin confirmacion). `actionInProgress = true` muestra "Processing...".
  4. `S` (mayuscula): `ConfirmDialog("Stop service X?")`. Confirmar: `serviceAction(name, 'stop')`.
  5. `R` (mayuscula): `ConfirmDialog("Restart service X?")`. Confirmar: `serviceAction(name, 'restart')`.
  6. Post-accion: `fetchServices()` automatico. Si hay error, se muestra en rojo bajo la lista.
  7. `r`: refresh manual.
* **Fricciones detectadas:**
  - Asimetria de confirmacion: `start` es inmediato, `stop` y `restart` requieren confirmacion. En un contexto de desarrollo, iniciar un servicio es raramente destructivo, pero la inconsistencia puede sorprender.
  - No hay indicador visual del servicio que esta siendo modificado durante `actionInProgress` (solo el texto generico "Processing...").
* **Puntos de abandono:** Ninguno critico.
* **Mensajes ambiguos:** "Processing..." es vago pero aceptable en una TUI de operaciones cortas.
* **Feedback del sistema suficiente:** Si
* **Tiempo a valor aceptable:** Si
* **Recuperacion ante error:** Si — `serviceActionError` visible bajo la lista.
* **Consistencia con el resto del producto:** Parcial (asimetria de confirmacion).

### Flujo: Profiles (Perfiles de instalacion — Pro)

* **Nombre:** Profiles — Crear, ver, importar, editar y eliminar perfiles
* **Objetivo del usuario:** Guardar el estado actual de paquetes instalados como un perfil reutilizable, o restaurar un perfil en otra maquina.
* **Inicio del flujo:** Pulsar `7` (requiere Pro; si no, muestra `UpgradePrompt`).
* **Final esperado:** El perfil se crea, importa, edita o elimina; la lista refleja el estado nuevo.
* **Pasos (crear perfil):**
  1. `fetchProfiles()` al montar. Spinner si loading.
  2. `n`: cambia a modo `create-name`. Se muestra `TextInput` para nombre.
  3. Submit nombre: cambia a modo `create-desc`. Se muestra `TextInput` para descripcion.
  4. Submit descripcion: `exportCurrent(name, desc)` captura los paquetes actuales. Vuelve a modo `list`.
* **Pasos (importar perfil):**
  1. Desde modo `list`, `Enter` en un perfil carga su detalle (modo `detail`).
  2. En modo `detail`, `i` inicia `startImport()`. Cambia a modo `importing`.
  3. El async generator instala formulae y casks en secuencia, emitiendo lineas al `ProgressLog`.
  4. Al terminar, se muestra "Import complete. Press any key." Cualquier tecla vuelve al modo `list`.
* **Pasos (eliminar perfil):**
  1. Desde modo `list`, `d` muestra `ConfirmDialog`.
  2. Confirmar: `deleteProfile()`. Vuelve al modo `list`.
* **Fricciones detectadas:**
  - En el modo `create-desc`, si `exportCurrent` falla, el `loadError` se muestra pero el flujo no retrocede al paso anterior ni permite corregir. El `finally { setMode('list') }` garantiza salida pero sin mensaje de exito o fracaso claro.
  - La importacion puede instalar decenas de paquetes sin indicar cuantos quedan. Solo se ve el output del stream.
  - El modo `importing` suprime teclas globales (modal abierto), pero no hay forma de cancelar una importacion en curso desde la UI (solo via cierre de la vista).
* **Puntos de abandono:**
  - Error silencioso en `exportCurrent` que devuelve al usuario a la lista sin saber si tuvo exito.
* **Mensajes ambiguos:** `profiles_importComplete: "Import complete. Press any key."` es claro. `profiles_exportHint` explica bien la accion de exportar.
* **Feedback del sistema suficiente:** Parcial — falta feedback explicito de exito en `exportCurrent`.
* **Tiempo a valor aceptable:** Si
* **Recuperacion ante error:** Parcial — importacion tiene manejo de errores; exportacion sale silenciosamente en error.
* **Consistencia con el resto del producto:** Si

## 6.3 Analisis transversal

### Consistencia de patrones de operacion destructiva

El patron `ConfirmDialog` (`y/Y` para confirmar, `n/N/Esc` para cancelar, modal abierto para suprimir teclas globales) se aplica de forma consistente en todas las acciones destructivas de la TUI. La unica excepcion menor es `start` de servicio (sin confirmacion, justificable). En BrewBar, la inconsistencia es mas notable: "Upgrade All" tiene `.confirmationDialog`, upgrade individual no.

### Consistencia en gestion del ProgressLog post-completado

| Vista | Esc limpia el log | Patron correcto |
|-------|-------------------|-----------------|
| InstalledView | Si (corregido en v0.2.0) | Si |
| OutdatedView | No — Esc navega fuera; `r` fetches datos pero la lista nunca se muestra | No |
| SearchView | No — Esc navega fuera | No |
| PackageInfoView | Si (Esc via goBack() es correcto en este contexto) | Si |
| SmartCleanupView | Via `r` (stream.clear + analyze) | Aceptable |
| HistoryView | ProgressLog inline; no bloquea la vista | Si |
| SecurityAuditView | ProgressLog inline; no bloquea la vista | Si |

### Consistencia de feedback de estado

Todos los flujos tienen `<Loading>` durante carga inicial y `<ErrorMessage>` en caso de error de fetch. El patron de `ProgressLog` + icono de resultado (checkmark/X) + hint de tecla es consistente en todos los flujos que ejecutan operaciones largas. La unica inconsistencia es el post-completado en OutdatedView y SearchView.

### Persistencia de estado entre navegaciones

Ninguna vista persiste su estado local (cursor, filtro, busqueda) al navegar fuera y volver. Esto es un compromiso conocido de la arquitectura `useState` por vista. El impacto es mayor en las vistas con filtros complejos (HistoryView, InstalledView) y menor en vistas simples (Doctor, Config). Para una TUI de uso frecuente, esta friccion es observable pero no bloqueante.

### Calidad del copy

Los textos en `en.ts` y `es.ts` son claros, concisos y apropiados para el contexto tecnico. Los hints de teclado en el footer son descriptivos. Los mensajes de confirmacion incluyen el nombre del elemento afectado. Los mensajes de error son explicativos (no "Something went wrong"). Los textos de BrewBar estan localizados correctamente en ingles y espanol via `String(localized:)` y String Catalog.

### Descubrimiento de vistas sin atajo numerico

`package-info` y `account` no tienen atajo de numero. `package-info` es correcto (vista dependiente de contexto). `account` podria beneficiarse de un atajo (actualmente solo accesible via Tab o historial de navegacion), especialmente para usuarios que necesitan activar o consultar el estado de licencia.

# 17. Localizacion / 18. Release readiness

> Auditor: release-auditor | Fecha: 2026-05-01

---

## Resumen ejecutivo

La infraestructura de localizacion de Brew-TUI es solida: el modulo i18n del TUI garantiza paridad completa en tiempo de compilacion (427 claves, en + es), y BrewBar usa String Catalog con variantes de plural correctas; no se detecta texto hardcodeado visible en produccion. Sin embargo, el estado de release presenta riesgos serios: BrewBar carece completamente de notarizacion, tres canales de distribucion publican versiones desincronizadas (Formula en 0.5.3, Cask en 0.1.0, jsr.json en 0.5.2 frente al 0.6.1 actual), el Cask apunta a un usuario GitHub incorrecto que produce HTTP 404, y el pipeline CI solo cubre Ubuntu/Node sin ningun paso Swift, codesign ni notarize.

## 17. Localizacion e internacionalizacion

### Checklist

* [x] **Strings externalizadas** — Todas las cadenas visibles del TUI se sirven a traves del modulo `src/i18n/`. BrewBar usa `Text(...)` (auto-extractado) y `String(localized:)` para contextos no-SwiftUI. No se detecta texto hardcodeado visible en produccion.
* [x] **Claves semanticas** — El TUI usa claves semanticas jerarquicas (ej. `view_dashboard`, `loading_fetchingBrew`, `sessions_emptyTitle`). BrewBar usa literales de texto como clave en el String Catalog, convencion estandar de Xcode aceptada.
* [x] **Plurales correctos** — TUI: `tp(baseKey, count)` con sufijos `_one`/`_other` en `en.ts`/`es.ts`. BrewBar: tres strings con plural rules en xcstrings (`%lld packages can be updated`, `%lld updates available`, `%lld vulnerable packages found`). No se detecta interpolacion `\(count)` fuera de plural rules.
* [x] **Fechas localizadas** — `src/utils/format.ts`: `formatDate()` usa `toLocaleDateString('es-ES')` o `toLocaleDateString('en-US')` segun locale detectado. `formatRelativeTime()` usa claves i18n. No se detectan `DateFormatter` con formato hardcodeado en Swift.
* [x] **Numeros y moneda localizados** — Los precios en `en.ts`/`es.ts` son literales de cadena contextuales (€9,95/mes, €82/ano). No se detecta `String(format: "%.2f")` para moneda en codigo de produccion.
* [x] **Layout soporta textos largos** — BrewBar: SwiftUI flexbox sin `frame(width:)` fijo en contenedores de texto criticos. TUI (Ink/flexbox): sin restricciones de ancho fijas sobre texto traducible.
* [ ] **RTL contemplado si aplica** — No aplica: `defaultKnownRegions` lista solo `en` y `es`, ambos LTR. No hay `.lproj` arabico ni hebreo. Marcado como no aplicable por diseno.
* [x] **No texto hardcodeado visible** — Las 14 entradas xcstrings sin traduccion ES (`"state": "new"` en una, sin bloque ES en las demas) pertenecen exclusivamente a `BrewBarDesignVariants.swift` envuelto en `#if DEBUG` y no llegan a builds de produccion.

### Hallazgos

| Elemento | Estado | Severidad | Evidencia | Accion |
|----------|--------|-----------|-----------|--------|
| Un string BrewBar con `state: new` en EN | Pendiente | Baja | `Localizable.xcstrings` linea 18: `"%@ security alert%@"` con metadato `"state": "new"` en bloque EN | Confirmar la traduccion ES en Xcode y marcar como revisada para evitar advertencias en Xcode 16 String Catalog validation |
| 14 claves xcstrings sin traduccion ES | Informativo | Informativo | `Localizable.xcstrings`: entradas `isCommentAutoGenerated: true` sin bloque `es:` | Todas proceden de `BrewBarDesignVariants.swift` (`#if DEBUG`); no afectan produccion. Envolver el fichero en `#if DEBUG` o moverlo a un target de development para eliminar el ruido |

## 18.1 Pre-release tecnico

### Checklist

* [x] **Build Release limpia** — No se detectan `TODO`/`FIXME`/`HACK` en rutas de produccion. No hay `fatalError()` sin guarda `#if DEBUG` en codigo Swift de produccion. No hay `try!` / `as!` fuera de tests. `print()` en TUI controlado por `src/utils/logger.ts` con `LOG_LEVEL`.
* [ ] **Archive correcto** — El xcarchive en disco es version 0.6.0 (creado 2026-04-30T20:17:03Z); `Project.swift` define 0.6.1. No existe `Makefile`, `Fastfile`, ni workflow GitHub Actions que automatice el paso de archive tras un bump de version.
* [x] **Firma correcta** — Release: `CODE_SIGN_STYLE: Manual`, `CODE_SIGN_IDENTITY: Developer ID Application`, `ENABLE_HARDENED_RUNTIME: YES`, `OTHER_CODE_SIGN_FLAGS: --timestamp`, team `GD6M44DYPQ` consistente. Debug: Automatic / Apple Development / Hardened Runtime NO (necesario para Xcode Previews). Configuracion correcta y deliberada.
* [x] **Assets correctos** — `Assets.xcassets/AppIcon.appiconset/Contents.json` completo: 16x16 a 512x512 en 1x y 2x. Todos los ficheros de imagen presentes. Sin `LaunchScreen` (LSUIElement: true — correcto para menubar app).
* [x] **Configuracion entorno correcta** — No se detectan URLs de debug, credenciales de test ni flags de desarrollo filtrando a Release. `#if DEBUG` guarda los caminos de desarrollo.
* [x] **Feature flags revisadas** — No existe sistema de feature flags externo (LaunchDarkly, Firebase). El gate de features Pro/Team es estatico en `src/lib/license/feature-gate.ts`; funciona correctamente en produccion.
* [x] **Logs verbosos eliminados o controlados** — TUI: `logger.ts` con `LOG_LEVEL` — `print()` desnudo solo en handlers CLI (`src/index.tsx`) donde es el canal de salida correcto. BrewBar Swift: `Logger(subsystem:category:)` de `os` framework con niveles apropiados; `CrashReporter.swift` usa `.info`/`.warning`. Sin volcado de secretos en logs.

## 18.2 Producto

### Checklist

* [ ] **Flujos criticos aprobados** — TUI: cobertura de tests unitarios en parsers, license manager, brew-api, stores. Sin tests de integracion end-to-end ni UI tests (`ink-testing-library` disponible pero no usado segun CLAUDE.md). BrewBar: target `BrewBarTests` definido en `Project.swift`, sin tests escritos.
* [x] **Bugs criticos resueltos** — No se detectan comentarios `// BUG:`, `// KNOWN ISSUE:`, `// WORKAROUND:` en rutas de produccion. No hay funcionalidad comentada que indique issues conocidos activos.
* [ ] **Crash-free threshold aceptable** — `CrashReporter.swift` es opt-in mediante UserDefaults; desactivado por defecto. Solo captura `NSException`; `fatalError` puro de Swift no es interceptado. Sin tipos de datos recopilados declarados.
* [x] **Accesibilidad minima validada** — BrewBar: `accessibilityLabel` presente en componentes de SwiftUI donde corresponde. TUI: terminal renderer (Ink) sin API de accesibilidad estandar — limitacion del entorno, no un deficit del producto.

## Checklist diferencial para el proximo release

Los siguientes pasos son necesarios para un release 0.6.1 (o siguiente) tecnicamente correcto. Ordenados por prioridad:

### Bloqueantes (Alta)

- [ ] **Notarizar BrewBar**: Ejecutar `xcrun notarytool submit BrewBar.zip --apple-id ... --team-id GD6M44DYPQ --wait` y `xcrun stapler staple BrewBar.app` antes de publicar el zip. Actualizar `exportOptions.plist` a `method: developer-id`.
- [ ] **Corregir Cask URL**: Cambiar `MoLinesGitHub` a `MoLinesDesigns` en `homebrew/Casks/brewbar.rb` y actualizar version a 0.6.1. Verificar con `brew audit --cask brewbar`.
- [ ] **Actualizar Homebrew Formula**: Actualizar URL en `homebrew/Formula/brew-tui.rb` a `brew-tui-0.6.1.tgz` y recalcular SHA256.
- [ ] **Actualizar jsr.json**: Sincronizar `"version"` a `0.6.1` antes de ejecutar `npx jsr publish`.
- [ ] **Anadir CI macOS**: Crear job en `.github/workflows/ci.yml` con `macos-latest` que compile BrewBar en Release para detectar regresiones Swift.

### Recomendados (Media)

- [ ] **Auditar entrada FileTimestamp**: Verificar si alguna dependencia del sistema usa timestamp de fichero. Eliminar la entrada de `PrivacyInfo.xcprivacy` si no hay justificacion o documentar el uso.
- [ ] **Automatizar publish-all.sh**: El script debe actualizar `jsr.json`, `Formula/brew-tui.rb` y `Casks/brewbar.rb` de forma no interactiva, incluyendo el paso de notarizacion de BrewBar.
- [ ] **Regenerar xcarchive**: Ejecutar `xcodebuild archive -scheme BrewBar -archivePath build/BrewBar.xcarchive` tras el bump de version para que el archivo en disco refleje 0.6.1.

### Mejoras (Baja)

- [ ] **Resolver xcstrings `state: new`**: Abrir Xcode, revisar `"%@ security alert%@"` en el String Catalog, confirmar traduccion ES y marcar como revisada.
- [ ] **Corregir `.gitignore` para exportOptions**: Cambiar `ExportOptions.plist` a `exportOptions.plist` (minuscula) o anadir ambas variantes para que el matching funcione en Linux CI.
- [ ] **Completar CHANGELOG**: Anadir entradas para 0.3.x, 0.4.0 y 0.6.0 o una nota de versiones internas.
